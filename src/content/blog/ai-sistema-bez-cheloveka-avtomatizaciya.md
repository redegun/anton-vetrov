Большинство людей думают, что AI — это чат. Задал вопрос — получил ответ. Но в [AI First архитектуре](/blog/arhitektura-ai-sistem-4-sloya/) промпт может прийти не только от человека. Крон, событие в CRM, триггер из другой системы — всё это источники промптов. **Система работает и без человека.**

Это принципиальное отличие [AI First подхода](/blog/chto-takoe-ai-first-podkhod/) от обычного внедрения. Обычный бот ждёт, когда человек напишет. AI First система работает 24/7, обрабатывает события, генерирует отчёты, мониторит показатели — автономно.

## Источники промптов без человека

В [4-слойной архитектуре](/blog/arhitektura-ai-sistem-4-sloya/) [провайдеры](/blog/provajdery-i-klienty-ai-sistemy/) формируют промпты. Не все провайдеры связаны с людьми:

### Кроны (расписание)

Запуск задач по расписанию — самый простой тип автономной работы:

- **06:00** — агент-аналитик собирает данные из CRM, формирует ежедневный отчёт
- **09:00** — агент-маркетолог проверяет метрики сайта, фиксирует аномалии
- **18:00** — агент продаж подводит итоги дня, обновляет прогноз
- **Каждые 30 минут** — агент мониторинга проверяет доступность сервисов

Крон-провайдер формирует промпт: «Время выполнить задачу X». Агент выполняет, создаёт [артефакт](/blog/ai-agent-s-pamyatyu-artefakty/) с результатом. Утром менеджер открывает дашборд и видит готовый отчёт, подготовленный ночью.

### События (триггеры)

Event-провайдер реагирует на события в системах:

**CRM-события:**
- Новый лид → агент продаж квалифицирует
- Сделка перешла в «Договор» → агент-документалист готовит черновик
- Лид не отвечает 3 дня → агент запускает follow-up
- Крупная сделка закрыта → агент-аналитик обновляет отчёт

**Почтовые события:**
- Входящее письмо → агент классифицирует и маршрутизирует
- Письмо от VIP-клиента → агент уведомляет менеджера и готовит контекст
- Автоматический ответ (OOO) → агент обновляет статус контакта

**Системные события:**
- Сервер превысил нагрузку → агент мониторинга создаёт инцидент
- Новая запись в базе знаний → агент переиндексирует для поиска
- Баланс API ниже порога → агент уведомляет администратора

### Внутренние процессы

Агенты могут запускать других агентов. Это цепочки обработки:

1. Крон запускает агента-аналитика → создаёт отчёт (артефакт)
2. Отчёт содержит аномалию → событие запускает агента по инцидентам
3. Агент по инцидентам анализирует причину → создаёт рекомендацию (артефакт)
4. Рекомендация требует действия → агент уведомляет менеджера

Человек подключается только на шаге 4, когда нужно принять решение. Шаги 1-3 — автономны.

## Практические сценарии

### Утренний запуск без людей

Рабочий день AI-системы начинается до прихода сотрудников:

**05:00** — Крон. Агент-аналитик собирает данные:
- Из CRM: новые лиды за ночь, изменения по сделкам
- Из Метрики: трафик, конверсии, источники
- Из почты: входящие письма
- Результат: артефакт «Утренняя сводка: 15 марта»

**05:30** — Цепочка. Агент продаж читает утреннюю сводку:
- Квалифицирует ночные лиды
- Определяет приоритеты на день
- Результат: артефакт «Приоритеты продаж: 15 марта»

**06:00** — Цепочка. Агент-документалист:
- Проверяет сделки в статусе «Ожидает документ»
- Генерирует черновики КП и договоров
- Результат: черновики в артефактах

**08:00** — Менеджер приходит на работу и видит:
- Готовую сводку за ночь
- Квалифицированные лиды с приоритетами
- Черновики документов для проверки

Экономия: 1.5-2 часа ежедневной рутины на каждого менеджера.

### Мониторинг 24/7

Агент мониторинга работает круглосуточно:

- Проверяет доступность сайтов и сервисов
- Мониторит балансы API-ключей
- Отслеживает аномалии в данных CRM
- Следит за конкурентами (новые страницы, цены)

При обнаружении проблемы — создаёт инцидент, уведомляет ответственного, предлагает решение. Ночью в 3:00 упал сайт? Агент заметит через 5 минут, создаст инцидент и разбудит дежурного.

### Автоматический follow-up

Лид не ответил? Агент знает, когда написать повторно:

- **День 1:** первое письмо
- **День 3:** нет ответа → follow-up с дополнительной ценностью
- **День 7:** нет ответа → последняя попытка с другим углом
- **День 14:** нет ответа → перевод в «холодные», уведомление менеджера

Каждое письмо персонализировано — агент использует [контекст](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/) из CRM, историю взаимодействий, артефакты о компании клиента.

## Безопасность автономной работы

Автономная система — это мощно, но требует контроля. Я закладываю несколько уровней защиты:

### Human-in-the-loop для критичных действий

Агент может автономно:
- Квалифицировать лидов
- Генерировать черновики документов
- Формировать отчёты и аналитику
- Отправлять стандартные ответы

Агент запрашивает подтверждение для:
- Отправки коммерческих предложений
- Изменения цен и условий
- Публикации контента
- Крупных финансовых операций

### Логирование и аудит

Все автономные действия логируются. Менеджер может посмотреть:
- Какие решения принял агент
- На основе какого контекста
- Какие артефакты создал
- Какие уведомления отправил

### [Приватные сессии](/blog/bezopasnost-ai-agentov-sessii/)

Автономные процессы работают в [приватных сессиях](/blog/bezopasnost-ai-agentov-sessii/) — полный доступ к данным, но изолированная среда. Внешний пользователь не может запустить автономный процесс или получить его результаты.

## Когда автономность не нужна

Не всё нужно автоматизировать. Автономная работа имеет смысл когда:

- **Задача повторяется** — ежедневные отчёты, регулярные проверки
- **Задача формализуема** — есть чёткие критерии и алгоритм
- **Скорость реакции критична** — мониторинг, обработка ночных заявок
- **Объём большой** — сотни лидов в день, десятки писем

Не автоматизируйте:
- Стратегические решения
- Нестандартные переговоры
- Первый контакт с VIP-клиентами
- Задачи, требующие человеческого суждения

## Как внедрить автономную работу

1. **Начните с кронов** — ежедневный отчёт, утренняя сводка. Минимальный риск, быстрый результат
2. **Добавьте события** — реакция на новые лиды, изменения статусов в CRM
3. **Настройте цепочки** — агенты запускают друг друга через артефакты
4. **Расширяйте постепенно** — больше триггеров, больше автоматических решений, меньше ручного контроля
