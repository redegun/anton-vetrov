У каждого бизнеса свои каналы: кто-то работает через Telegram, кто-то через почту, кто-то через CRM-виджет. Когда ко мне приходят с задачей [внедрения ИИ](/services/ai-vnedrenie-pod-klyuch/), первый вопрос: «А к Telegram подключите? А к почте? А к Bitrix24?»

Подключу. И через месяц добавлю WhatsApp. А через два — голосового ассистента. Потому что в [AI First архитектуре](/blog/arhitektura-ai-sistem-4-sloya/) каналы — это плагины, а не монолитные интеграции.

Внешний слой архитектуры — провайдеры и клиенты. Это точка входа в AI-систему и точка выхода. Разберём, как они устроены.

## Провайдеры: адаптеры к внешнему миру

Провайдер — это адаптер, который соединяет внешнюю систему с AI-ядром. Снаружи — протокол целевой программы (Telegram API, IMAP, Bitrix24 API). Внутри — стандартный протокол системы.

Задача провайдера — **сформировать первоначальный промпт**. Когда клиент пишет в Telegram, провайдер:

1. Принимает сообщение через Telegram API
2. Извлекает метаданные: кто написал, из какого чата, время, тип сообщения
3. Формирует стандартный промпт, понятный системе
4. Передаёт в сессию

Провайдер — это переводчик. Он переводит с языка Telegram на язык AI-системы. И с языка AI-системы — обратно на язык Telegram.

### Виды провайдеров

**Мессенджеры:**
- Telegram — самый популярный канал для бизнеса в России
- WhatsApp Business — для клиентов, предпочитающих WhatsApp
- Viber — для отдельных сегментов аудитории

**Почта:**
- IMAP-провайдер — слушает входящие письма
- SMTP-провайдер — отправляет ответы
- Может парсить вложения (PDF, документы) и включать их в контекст

**CRM-системы:**
- Bitrix24-провайдер — слушает события (новый лид, смена статуса, комментарий)
- amoCRM-провайдер — аналогично для amoCRM
- Подробнее: [интеграция ИИ с CRM](/services/ai-integraciya-crm/)

**Веб:**
- Webhook-провайдер — принимает HTTP-запросы от любых систем
- WebSocket-провайдер — двусторонняя связь в реальном времени
- Виджет на сайте — чат для посетителей

**Внутренние:**
- Cron-провайдер — запускает задачи по расписанию
- Event-провайдер — реагирует на внутренние события системы
- Подробнее: [AI без человека](/blog/ai-sistema-bez-cheloveka-avtomatizaciya/)

## Клиенты: приложения для управления

Клиенты — это самостоятельные приложения, которые работают с AI-системой. В отличие от провайдеров, клиенты:

- Знают протокол подключения
- Могут не только формировать промпт, но и **управлять системой**
- Предоставляют полезную функцию пользователю

### Примеры клиентов

**CLI (командная строка)** — для разработчиков и администраторов. Управление агентами, мониторинг, отладка.

**Дашборд** — для менеджеров. Визуализация работы системы: сколько обращений обработано, какие агенты работают, статистика по каналам.

**Мобильное приложение** — для руководителей. Уведомления о важных событиях, быстрые решения, аналитика.

## Унификация протоколов

Ключевое преимущество провайдеров: внутри системы — единый стандартный протокол. Неважно, откуда пришёл промпт — из Telegram, почты или CRM. Для [агента](/blog/cifrovye-rabotniki-ai-agenty/) все запросы выглядят одинаково.

Это означает:

1. **Агент не знает** про Telegram, почту или CRM — он работает с абстрактным промптом
2. **Один агент** может обслуживать несколько каналов — не нужно создавать отдельного бота для каждого мессенджера
3. **Новый канал** = новый провайдер. Агенты, [артефакты](/blog/ai-agent-s-pamyatyu-artefakty/), [контекст-инжиниринг](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/) — всё остаётся как было

### Пример

Агент продаж квалифицирует лидов. Лид может прийти из:
- Telegram (провайдер формирует промпт из сообщения)
- Почты (провайдер парсит письмо и формирует промпт)
- CRM (провайдер ловит событие «новый лид» и формирует промпт)
- Формы на сайте (webhook-провайдер принимает заявку)

Для агента — это один и тот же процесс квалификации. Разные точки входа, одна логика обработки.

## Формирование первоначального промпта

Провайдер не просто передаёт текст сообщения. Он формирует структурированный промпт с метаданными:

```
Источник: Telegram
Пользователь: Иван Петров (@ivanpetrov)
Время: 2026-03-15 10:30 MSK
Тип: текстовое сообщение
Контекст: группа «Запросы клиентов»

Сообщение:
Здравствуйте, интересует автоматизация отдела продаж.
У нас 15 менеджеров, работаем в Bitrix24.
```

Метаданные важны для [контекст-инжиниринга](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/): агент знает, из какого канала пришёл запрос, и может адаптировать стиль ответа.

## Провайдеры и рабочий цикл

В [4-слойной архитектуре](/blog/arhitektura-ai-sistem-4-sloya/) рабочий цикл замкнут на сессию. Провайдер участвует только в начале (формирует промпт) и в конце (отдаёт ответ):

1. **Провайдер →** формирует промпт из внешнего события
2. **Сессия →** обогащает историей
3. **Агент →** добавляет SOUL.md, скилы
4. **Артефакты →** подмешивают знания
5. **Ядро →** маршрутизирует в [LLM](/blog/kak-vybrat-llm-dlya-biznesa/), получает ответ
6. **← Провайдер** отдаёт ответ в нужном формате (Telegram-сообщение, письмо, CRM-комментарий)

## Как подключить провайдер

Благодаря [плагинной архитектуре](/blog/plaginnaya-arhitektura-ai/) подключение нового провайдера — предсказуемый процесс:

1. **Определите канал** — какую внешнюю систему подключаем
2. **Реализуйте адаптер** — приём сообщений + формирование промпта
3. **Настройте обратный канал** — как отправлять ответы
4. **Определите метаданные** — что передавать в промпте
5. **Настройте [безопасность](/blog/bezopasnost-ai-agentov-sessii/)** — приватная или публичная сессия
6. **Зарегистрируйте в ядре** — провайдер начинает работать

Типичное время: 3-5 дней на провайдер. Не месяцы.

## Мультиканальность без хаоса

Когда у вас 5-7 каналов без архитектуры — это хаос. Каждый канал — отдельный бот с отдельной логикой. Обновить поведение = обновить 7 ботов.

С провайдерами: один агент, семь провайдеров. Обновили скил квалификации лидов = обновилось во всех каналах одновременно. Обновили SOUL.md агента = стиль общения изменился везде.
