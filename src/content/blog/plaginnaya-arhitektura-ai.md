Самая частая проблема AI-проектов — они не масштабируются. Подключили чат-бота к Telegram — работает. Нужен бот для почты — пишем с нуля. Нужна интеграция с CRM — ещё один проект. Каждый новый канал или функция — это месяцы разработки.

Решение: **микроядро + плагины**. Каждый элемент AI-системы организован как подключаемый модуль. Новые каналы, хранилища, инструменты добавляются без изменения ядра.

Это не теория — это архитектурный принцип [AI First подхода](/blog/chto-takoe-ai-first-podkhod/), который я закладываю при [проектировании AI-архитектуры](/services/ai-arhitektura/).

## Принцип микроядра

Микроядро — это минимальное ядро системы, которое отвечает за:
- Управление сессиями
- Формирование [контекста](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/)
- Рабочий цикл (промпт → обработка → результат)
- Регистрацию и загрузку плагинов

Всё остальное — плагины. Провайдеры, артефакты, инструменты, LLM-подключения — всё подключаемое.

Аналогия: операционная система. Ядро Linux отвечает за базовые функции. Драйверы (камера, звук, сеть) — это модули. Новое устройство — новый драйвер, а не новая ОС.

## Что является плагином

### Провайдеры

[Провайдеры](/blog/provajdery-i-klienty-ai-sistemy/) — адаптеры к внешним системам. Каждый провайдер — отдельный плагин:

- **Telegram-провайдер** — принимает сообщения из Telegram, формирует промпт
- **Email-провайдер** — парсит входящие письма, формирует промпт
- **CRM-провайдер** — слушает события в CRM (новый лид, смена статуса)
- **Календарь-провайдер** — отслеживает события календаря
- **Webhook-провайдер** — принимает HTTP-запросы от любых систем

Сегодня у вас Telegram и почта. Завтра нужен WhatsApp — подключаете провайдер, а не переписываете систему. Послезавтра голосовой ассистент — ещё один провайдер.

### Хранилища артефактов

[Артефакты](/blog/ai-agent-s-pamyatyu-artefakty/) могут храниться по-разному, и тип хранения — тоже плагин:

- **File-storage** — MD-файлы на диске (простой, надёжный)
- **Vector-storage** — векторная база данных (масштабируемый, семантический поиск)
- **In-memory** — оперативная память (быстрый, для кэша)
- **Database** — SQL/NoSQL база (структурированный, для отчётности)

Можно комбинировать: ежедневные отчёты в MD-файлах, база знаний в векторном хранилище, кэш в памяти. Каждый плагин хранения регистрируется в ядре и работает параллельно.

### Инструменты (Tools)

Каждый инструмент — плагин:

- **CRM-tool** — создание/чтение/обновление задач и сделок
- **Email-tool** — отправка писем
- **Calendar-tool** — управление событиями
- **File-tool** — работа с файлами
- **Search-tool** — поиск в базе знаний

### Skills

Skills — MD-файлы с инструкциями. Тоже подключаемые:

- **Квалификация лидов** — как оценить потенциал клиента
- **Подготовка КП** — структура и стиль коммерческого предложения
- **Анализ продаж** — метрики и формат отчёта

Новый навык = новый MD-файл. Никакого кода.

### LLM-провайдеры

[Модели](/blog/kak-vybrat-llm-dlya-biznesa/) подключаются как плагины:

- **Claude-provider** — Claude API (Haiku, Sonnet, Opus)
- **OpenAI-provider** — GPT API (4o, 4o-mini)
- **Local-provider** — локальная модель (Llama, Mistral)

Маршрутизация между ними — часть ядра, но сами модели — плагины. Вышла новая модель — подключил провайдер, настроил правила маршрутизации, система использует.

## Преимущества плагинной архитектуры

### Простота

Каждый плагин отвечает за свою зону. Telegram-провайдер не знает про CRM. CRM-tool не знает про векторное хранилище. Границы чёткие, код понятный, баги изолированные.

### Расширяемость

Новый канал — новый провайдер. Новый инструмент — новый tool. Новая модель — новый LLM-провайдер. Ядро не меняется.

Это критично для бизнеса: вы не платите за «переделку всего» каждый раз, когда нужна новая функция. Вы платите за один плагин.

### Разделение обязанностей

Команда может работать параллельно: один разработчик делает Telegram-провайдер, другой — CRM-tool, третий — skill для квалификации лидов. Они не мешают друг другу, потому что работают с разными плагинами.

### Безопасность

[Приватные и публичные сессии](/blog/bezopasnost-ai-agentov-sessii/) — архитектурный принцип, реализованный через плагины. Каждый плагин имеет свой уровень доступа. Публичная сессия не может загрузить приватный плагин хранения.

### Тестируемость

Каждый плагин тестируется независимо. Сломался CRM-tool? Остальная система работает. Обновили Telegram-провайдер? Проверяем только Telegram, остальное не трогаем.

## Как спроектировать плагинную архитектуру

### Шаг 1: Определите ядро

Что минимально необходимо? Управление сессиями, [контекст-инжиниринг](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/), рабочий цикл, регистрация плагинов. Всё остальное — выносим в плагины.

### Шаг 2: Определите интерфейсы плагинов

Каждый тип плагина имеет стандартный интерфейс:
- Провайдер: `onMessage(raw) → prompt`
- Хранилище: `save(artifact)`, `load(query) → artifacts`
- Tool: `execute(params) → result`
- LLM: `generate(context) → response`

Стандартные интерфейсы — это контракты. Любой плагин, который соответствует контракту, подключается автоматически.

### Шаг 3: Определите текущий набор плагинов

Какие провайдеры нужны сейчас? Какие инструменты? Какие модели? Начните с минимального набора и расширяйте по мере необходимости.

### Шаг 4: Планируйте расширение

Какие каналы добавите через 3 месяца? Какие инструменты? Это не обязательно, но помогает проверить: архитектура действительно расширяемая, или вы заложили ограничения?

## Пример: от MVP до полной системы

**Месяц 1 (MVP):**
- Провайдеры: Telegram
- Хранилище: MD-файлы
- Tools: CRM (чтение)
- LLM: Claude Sonnet
- Агенты: 1 (продажи)

**Месяц 3:**
- + Провайдеры: Email, Webhook
- + Хранилище: Векторное (база знаний)
- + Tools: CRM (запись), Email (отправка)
- + LLM: GPT-4o-mini (быстрые ответы)
- + Агенты: 3 (продажи, поддержка, аналитика)

**Месяц 6:**
- + Провайдеры: WhatsApp, Голосовой ассистент
- + Хранилище: SQL (отчётность)
- + Tools: Календарь, Документы
- + LLM: Локальная Llama (приватные данные)
- + Агенты: 6 (+ документалист, HR, маркетинг)

Каждый шаг — добавление плагинов, а не переделка системы. Инвестиции растут линейно, а не экспоненциально.

## Связь с другими принципами AI First

Плагинная архитектура — это реализация [4-слойной модели](/blog/arhitektura-ai-sistem-4-sloya/):
- Провайдеры (1 слой) → плагины
- Сессии (2 слой) → ядро
- Артефакты (3 слой) → плагины хранения
- Инструменты и LLM (4 слой) → плагины

[Контекст-инжиниринг](/blog/kontekst-inzhiniring-vazhneje-vybora-modeli/) работает поверх плагинов — ядро собирает контекст из разных плагинов и формирует единый запрос для LLM.

[Маршрутизация LLM](/blog/kak-vybrat-llm-dlya-biznesa/) — часть ядра, но LLM-провайдеры — плагины. Новая модель подключается без изменения логики маршрутизации.

## Итог

Плагинная архитектура — это не усложнение, а упрощение. Микроядро + плагины позволяют масштабировать AI-систему предсказуемо и экономично. Новый канал, инструмент или модель — это один плагин, а не переделка всего.
